## 自分のアプリをリファクタリング

このアプリは自分が実務未経験時に作ったアプリです。

自分が過去に書いたコードを振り返るのって結構つらいところがあって、逆によく動くアプリが作れたなと驚きすらあります。

できれば見返したくはなかったですが、リーダブルコードを読むにあたってこれ以上の題材はないと思ったので、振り返る事にしました。

恐らく、リファクタリング関連の書籍が出るたびに自分が初めて作成したアプリをこすり倒す事になると思いますがまずは第一弾です。

### 前提

#### 対象としたアプリケーション

運転日報管理アプリ
<https://github.com/hiroyuki9614/odo_track>

題してODO TRACKです。

これは距離計のODO Meterと計測のTrackを掛けた単語なのですが、もしかしたらひらめいたタイトルなんですかね。

#### スタック

rails v7.1.2
vue.js v3.4.1

フロントエンドはJavaScript, バックエンドはRubyで書いています。

#### 対象とするコード

RubyとJavaScriptで、特に酷いコードを対象としてピックアップしていきます。

あと、あくまでリーダブルコードの各章に沿った内容で修正を行うのでそもそものコードが汚くても問題として取り上げている箇所以外は修正しません。

### 第二章 名前に情報を詰め込む

#### 明確な単語を選ぶ

もっと変な命名がざくざく出てくるかと思いきや、意外と雑な命名が出てこなくて全然おもしろくなかったのですが、思ったよりも出てこなくて驚きました。

今ほどは性能が良くなかったとはいえ、AIが普及し始めてたから・・・？

あまりいい感じのコードがなかった章は自分で見返すようでタイトルのみ記入。

#### 明確な単語を選ぶ

#### letやretvalなどの汎用的な名前を避ける

#### 抽象的な名前よりも具体的な名前を使う

#### 名前に情報を追加する

変数は短いコメントのようなものなので、何のための変数かを明示するとコードがより説明的になる。

例えば、このコードは文脈からしてバリデーションチェックの結果を返す関数だとわかりますが、関数名がisValidだと何のバリデーションチェックの結果なのかがわからない。

コードを見る限り、他の関数を走らせた結果、全ての入力項目がバリデーションを通過した場合のみtrueを返す関数のようす。

```javascript
const isValid = () => {
 if (departureDatetimeIsValid.value && arrivalDatetimeIsValid.value && departureLocationIsValid.value && arrivalDatetimeIsValid.value &&
  departureLocationIsValid.value && arrivalLocationIsValid.value && noteIsValid.value && alcoholCheckIsValid.value) {
  return true;
 } else {
  return false;
 }
};
```

```javascript

  window.scrollTo({

```

このコードもコメントアウトは打っているが、そもそも変数を説明的にすればコメントアウトも不要になる。

```javascript
const scrollToTop = () => {
 //
}
```

あと、この変数名は邪悪だった。

```javascript
const scTimer = ref(0);
const scY = ref(0);
```

これはスクロールという意味だった。

```javascript
const scrollTimer = ref(0);
const scrollY = ref(0);
```
